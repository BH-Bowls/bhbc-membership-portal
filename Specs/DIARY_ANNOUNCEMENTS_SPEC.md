## 1. Overview

### 1.1 Diary Panel

A personalised "Coming Up" panel on the home page showing the logged-in member's upcoming
duties and games, in chronological order, with no upper date cutoff. Items with no future
occurrences are simply absent.

**Data sources (all future dates only, relative to today):**

| # | Source | What it shows |
|---|--------|---------------|
| 1 | `CleaningRota` sheet (MEMBERS spreadsheet) | Member's next cleaning duty dates |
| 2 | `SweepingRota` sheet (MEMBERS spreadsheet) | Member's next sweeping duty dates |
| 3 | `Games` sheet (FRIENDLIES spreadsheet) | Games where member is `tea_lead`, `tea_first`, or `tea_second` |
| 4 | `Games` + `Players` sheets (FRIENDLIES spreadsheet) | Friendlies the member has entered (status O, X, S only — not P/C/A) |
| 5 | Competitions sheets (COMPETITIONS spreadsheet) | Member's Pending matches where `played_date` has been set |
| 6 | `AvailabilityEvents` + `AvailabilityResponses` sheets (AVAILABILITY spreadsheet) | Open events the member hasn't responded to yet (nudge) |
| 7 | `AvailabilityEvents` + `AvailabilityResponses` sheets (AVAILABILITY spreadsheet) | Concluded events where member said Yes (confirmed date in diary) |
| 8 | Competitions sheets (COMPETITIONS spreadsheet) | Singles matches where the member is the assigned `marker`, `played_date` is set and >= today |

### 1.2 Announcements

Club-wide notices displayed above the diary on the home page. Multiple announcements can be
live simultaneously. Each has a message body, an expiry datetime, and metadata.

- **Who can create/edit/delete:** Admin, Captain, GMC
- **Who can view:** All logged-in members
- **Expired announcements:** Hidden from home page but kept in the sheet. Only visible in the
  admin UI (greyed out), where they can be deleted.

## 2. New Google Sheet — HomeAnnouncements

Add a new tab called `HomeAnnouncements` to the **CONFIG spreadsheet**
(`PORTAL_CONFIG_SPREADSHEET_ID`).

### 2.1 Sheet Setup

Create the sheet with these **exact** column headers in row 1:

| Col | Header (exact) | Normalised key | Type | Notes |
|-----|----------------|----------------|------|-------|
| A | `ID` | `id` | String | UUID v4 generated by the app on create |
| B | `Message` | `message` | String | Announcement body text |
| C | `Expires At` | `expires_at` | String | ISO datetime string e.g. `2026-05-25T23:59:00.000Z` |
| D | `Created By` | `created_by` | String | FK → `Members.user_name` |
| E | `Created At` | `created_at` | String | ISO datetime string |
| F | `Updated By` | `updated_by` | String | FK → `Members.user_name` (nullable) |
| G | `Updated At` | `updated_at` | String | ISO datetime string (nullable) |

Range: `A2:G`. Use `getColumnMap()` for all column access — do not hardcode positions.

### 2.2 ID Generation

Use `crypto.randomUUID()` (available in Node 18+) to generate the `id` on creation.
No read-before-write needed — UUIDs are collision-safe.

## 5. Data Layer — src/lib/diary-sheets.ts

Create `src/lib/diary-sheets.ts`. This file contains all Sheets reads needed to build the
diary for a given user. Use `batchGet` per spreadsheet to minimise API calls.

### 5.1 MEMBERS spreadsheet batchGet

Single `batchGet` call fetching:
- `CleaningRota!A2:E` — all cleaning rota rows
- `SweepingRota!A2:C` — all sweeping rota rows

**CleaningRota parsing:**
- Columns are positional: A=date, B=lead, C=second, D=third, E=fourth
- The `date` column is a freeform display string like `"Sat, 05 September"` — it does NOT
  contain a year. To determine if it is in the future, parse day+month and assume the current
  or next occurrence of that month/day. Use this logic:
  - Parse the display string to extract month name and day number
  - Build a candidate date for the current year
  - If that date is in the past, add 1 year
  - Use `parseUKDate()` / date helpers from `src/lib/date-utils.ts` where possible
- Filter rows where the member's `userName` appears in any of the four cleaner columns
- Return matching rows as diary items with `type: 'cleaning'`

**SweepingRota parsing:**
- Columns are positional: A=date, B=user_name, C=is_blocked
- The `date` column is stored as a Google Sheets serial number (integer days since 30 Dec 1899)
  — convert to JS Date using: `new Date((serialNumber - 25569) * 86400 * 1000)`
- Skip rows where `is_blocked` is `TRUE`
- Filter rows where `user_name` matches the member's `userName`
- Only include rows where the parsed date is >= today
- Return as diary items with `type: 'sweeping'`

### 5.2 FRIENDLIES spreadsheet batchGet

Single `batchGet` call fetching:
- `Games!A2:ZZ` — all games rows

**Tea duty detection:**
- For each game row, check if the member's `userName` matches `tea_lead`, `tea_first`,
  or `tea_second` columns
- Only include games where status is `O`, `X`, or `S` (open, selecting, selected/published)
  — not `P` (played), `C` (cancelled), or `A` (abandoned)
- Only include games where the game date is >= today
- Use `parseNormalizedDate()` from `src/lib/date-utils.ts` to parse the `date` column
- `subLabel` should reflect the role: `"Tea Lead"`, `"Tea (Helper)"` as appropriate
- Return as diary items with `type: 'tea'`, `linkUrl: '/friendlies'`

**Friendly entries detection:**
- The `Players` sheet has a dynamic structure: fixed left columns then one column per game
  (column header = tab_name). Do NOT read the entire Players sheet — it could be very wide.
- Instead, use the `Games` sheet data already fetched: look for games where status is
  `O`, `X`, or `S` and date >= today.
- For each such game, check if the member has entered by calling the existing
  `getPlayerEntryStatus()` function or equivalent from `src/lib/friendlies-sheets.ts`
  if it exists, OR read just the member's row from the Players sheet using their row number.
- **Important:** Do not re-read the entire Players sheet per game. Read the member's single
  row from `Players` once (find it by username in the first column), then check the relevant
  game columns in that row.
- Only include games where the member's entry status is `E` (entered), `S` (selected),
  or `R` (reserve) — not withdrawn or absent
- Return as diary items with `type: 'friendly'`, `label` = "vs {clubName} ({H/A})",
  `linkUrl: '/friendlies'`

### 5.3 COMPETITIONS spreadsheet batchGet

The competitions spreadsheet has 12 per-competition match sheets. Read all of them in a
single `batchGet` call. The sheet names are defined in `COMP_SHEET_CONFIG` in the existing
competitions data layer — read that config from the existing code, do not hardcode.

For each match sheet:
- Filter rows where `side1` or `side2` contains the member's `userName` (pipe-separated)
- Filter rows where `status` is `Pending` (not Complete, Walkover, Bye)
- Filter rows where `played_date` is not empty
- Filter rows where `played_date` (YYYY-MM-DD) is >= today
- Also collect rows where `marker` matches the member's `userName`, `played_date` is set and >= today, and `status` is `Pending` — these become `type: 'marker'` diary items (see §5.5 below)
- For the opponent display: the other side's usernames are pipe-separated — resolve to
  display names if possible, or show "opponent TBD" if side2 is empty/bye
- `label` = competition display name + round e.g. "Handicap Cup — R1"
- `subLabel` = "vs {opponentName}" or "vs TBD"
- `linkUrl` = `/competitions/{compId}`
- Return as diary items with `type: 'competition'`

### 5.4 Marker duty (from COMPETITIONS batchGet — same call as §5.3)

This uses the data already fetched in §5.3 — no additional API call.

For each match row across all competition sheets:
- Check if the member's `userName` matches the `marker` column
- Only include rows where `marker` equals `userName` (exact match — single username, not
  pipe-separated)
- Only include rows where `played_date` is set (non-empty) and >= today
- Only include rows where `status` is `Pending`
- Resolve player names: `side1` and `side2` are pipe-separated usernames — for a singles
  match each side has exactly one player. Resolve both to display names for the subLabel.
  If a name cannot be resolved, use the username as fallback.
- `label` = competition display name + round e.g. "Handicap Cup — R1"
- `subLabel` = "Marking: {side1PlayerName} vs {side2PlayerName}"
- `date` = `played_date` value (YYYY-MM-DD)
- `linkUrl` = `/competitions/{compId}`
- Return as diary items with `type: 'marker'`

---

### 5.5 AVAILABILITY spreadsheet batchGet

Single `batchGet` call fetching:
- `AvailabilityEvents!A2:ZZ` — all events
- `AvailabilityResponses!A2:ZZ` — all responses

**Availability nudge (open events, no response yet):**
- Find events where `status` = `Open`
- Find events where the member is an invitee — check `AvailabilityInvitees` sheet for
  member's `userName` linked to the event, OR if event is public (`group_id` is empty)
  include all members
- Cross-reference with `AvailabilityResponses` — if the member has NO response row for
  this event, it is a nudge item
- `label` = event title
- `subLabel` = "Awaiting your response"
- `linkUrl` = `/availability/{eventId}`
- `date` = event `created_at` date (used for sort order — nudges sort by creation date)
- Return as diary items with `type: 'availability_nudge'`

**Availability confirmed (concluded events, member said Yes):**
- Find events where `status` = `Concluded` and `concluded_slot_id` is not empty
- Find the winning slot from `AvailabilitySlots` — the slot whose `id` = `concluded_slot_id`
- Check `AvailabilityResponses` — if member responded `Yes` to that slot
- Only include if the concluded slot's datetime is >= today
- `label` = event title
- `subLabel` = "Confirmed — you said Yes"
- `linkUrl` = `/availability/{eventId}`
- `date` = concluded slot datetime (YYYY-MM-DD portion)
- Return as diary items with `type: 'availability_confirmed'`

### 5.6 Assembly and sorting

```typescript
export async function getDiaryItems(userName: string): Promise<DiaryItem[]>
```

This function:
1. Calls all batchGet operations above (MEMBERS, FRIENDLIES, COMPETITIONS, AVAILABILITY)
2. Combines all returned DiaryItem arrays
3. Sorts the combined list ascending by `date` (ISO string sort is safe for YYYY-MM-DD)
4. Returns the sorted array

## 7. API Routes

### 7.1 GET /api/diary

**File:** `app/api/diary/route.ts`

**Auth:** Session required (all authenticated members). Return 401 if no session.

**Logic:**
1. Check `getDiaryCache(userName)` — if hit, return cached data immediately
2. On cache miss: call `getDiaryItems(userName)` from `diary-sheets.ts`
3. Store result via `setDiaryCache(userName, items)`
4. Return `{ items }` as JSON

**Response shape:** `DiaryResponse` type

---

### 7.2 GET /api/announcements

**File:** `app/api/announcements/route.ts`

**Auth:** Session required. Return 401 if no session.

**Logic:**
1. Check `getAnnouncementCache()` — if hit, return cached data immediately
2. On cache miss: call `getActiveAnnouncements()` from `announcements-sheets.ts`
3. Store result via `setAnnouncementCache(announcements)`
4. Return `{ announcements }` as JSON

**Response shape:** `AnnouncementsResponse` type

---

### 7.3 GET /api/admin/announcements

**File:** `app/api/admin/announcements/route.ts`

**Auth:** Session required. Role must be `Admin`, `Captain`, or `GMC`. Return 403 otherwise.

**Logic:**
1. Call `getAllAnnouncements()` — returns all rows including expired. Do NOT use cache here;
   admin always sees fresh data.
2. Return `{ announcements }` as JSON

**Response shape:** `AdminAnnouncementsResponse` type

---

### 7.4 POST /api/admin/announcements

**File:** `app/api/admin/announcements/route.ts` (same file as GET above)

**Auth:** Session required. Role must be `Admin`, `Captain`, or `GMC`.

**Request body:**
```typescript
{
  message: string;    // required, non-empty
  expiresAt: string;  // required, ISO datetime string, must be in the future
}
```

**Validation:**
- `message` must be a non-empty string
- `expiresAt` must be a valid ISO datetime string and must be > now
- Return 400 with descriptive error if either fails

**Logic:**
1. Validate inputs
2. Call `createAnnouncement(message, expiresAt, session.user.userName)`
3. Call `clearAnnouncementCache()` so next home page load gets the new announcement
4. Return `{ success: true, announcement }` as JSON

---

### 7.5 PATCH /api/admin/announcements/[id]

**File:** `app/api/admin/announcements/[id]/route.ts`

**Auth:** Session required. Role must be `Admin`, `Captain`, or `GMC`.

**Request body:**
```typescript
{
  message: string;
  expiresAt: string;
}
```

**Validation:** Same as POST above.

**Logic:**
1. Validate inputs
2. Call `updateAnnouncement(id, message, expiresAt, session.user.userName)`
3. Call `clearAnnouncementCache()`
4. Return `{ success: true }` as JSON
5. If `updateAnnouncement` throws (id not found), return 404

---

### 7.6 DELETE /api/admin/announcements/[id]

**File:** `app/api/admin/announcements/[id]/route.ts` (same file as PATCH above)

**Auth:** Session required. Role must be `Admin`, `Captain`, or `GMC`.

**Logic:**
1. Call `deleteAnnouncement(id)`
2. Call `clearAnnouncementCache()`
3. Return `{ success: true }` as JSON
4. If `deleteAnnouncement` throws (id not found), return 404

## 9. Announcements Admin Page — app/admin/announcements/page.tsx

**Route:** `/admin/announcements`

### 9.1 Page Layout

- Page title: "Manage Announcements"
- Back link to wherever is appropriate (use `<RouterBackLink>`)
- "New Announcement" button — opens inline form below the button (not a modal)
- List of all announcements (active first, then expired), each as a card

### 9.2 New / Edit Announcement Form

Shown inline (not a modal). Fields:

| Field | Input type | Notes |
|-------|------------|-------|
| Message | `<textarea>` | Required. Multi-line. |
| Expires on | `<input type="date">` | Required. |
| Expires at | `<input type="time">` | Optional. Defaults to `23:59` if left blank. |

On save (POST for new, PATCH for edit):
- Combine date + time into ISO string: if time is blank use `23:59:00` local time converted
  to UTC. Use `new Date(dateStr + 'T' + timeStr)` and call `.toISOString()`.
- Validate that the combined datetime is in the future before submitting — show inline error
  if not, do not call the API
- On success: close the form, refresh the list
- On error: show inline error message — never use `alert()`

### 9.3 Announcement Cards

Each card shows:
- Message text
- Expiry: "Expires: Fri 23 May 2026 at 11:59pm"
- Created by + created date
- If expired: greyed out with a clear "Expired" badge
- If active: normal styling with a green "Active" badge

Action buttons on each card:
- **Edit** — populates the inline form with current values, switches POST to PATCH on save
- **Delete** — shows an inline confirmation ("Are you sure? This cannot be undone") before
  calling DELETE. Do not use `window.confirm()` — use inline state to show/hide the
  confirmation prompt.

## 15. Coding Standards Reminders

These are the most commonly missed rules in this codebase — check each one before
submitting any file:

- **File header comment** — every file starts with full path + one-line description
- **Every function, loop, if-statement, API call, and state update has a comment**
- **No `?.` optional chaining** — use explicit `if` checks
- **No `??` nullish coalescing** — use explicit `if` fallback blocks
- **No `.find()` or `.findIndex()`** — use explicit `for` loops
- **No `alert()`** — inline error state (`useState<string | null>`) only
- **No `window.confirm()`** — inline confirmation state only
- **Text contrast** — `text-gray-700` minimum for all readable text, never `text-gray-400/500/600`
- **Buttons and badges** — use `getButtonClasses()` and `getBadgeClasses()` from `theme-helpers.ts`
- **Dates from Sheets** — always use `parseUKDate()` / `parseNormalizedDate()` from
  `date-utils.ts`. Never call `new Date(sheetDateString)` directly except for ISO strings
  written by the app itself
- **Column access** — use `getColumnMap()`. CleaningRota and SweepingRota are positional
  exceptions (documented in SCHEMA.md)
- **Google Sheets calls** — always go through data-layer functions. Never call `googleapis`
  directly from route handlers
- **Env vars** — use named getter functions, never inline `process.env` in data-layer code
- **`session.user.userName`** — always the correct user identifier, including during impersonation
- **sessionStorage caching** — follow the exact pattern in CODING_STANDARDS.md §25
- **New protected routes** — add to `middleware.ts`, not guarded only inside the page
